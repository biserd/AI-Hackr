You are improving AIHackr, a SaaS AI-stack intelligence and competitor-monitoring product.

Current problem:
AIHackr can scan a SaaS website and produce an AI intelligence report, but the product risks feeling like a one-off curiosity tool. Many SaaS apps hide AI calls server-side, behind login, behind Cloudflare, through backend APIs, or inside authenticated product flows. If the scanner only says “No public signal,” users may not feel enough value to pay.

Goal:
Improve both:
1. The actual AI scanning and detection process.
2. The product’s path to PMF by turning AIHackr into competitive intelligence for AI-native SaaS.

Primary positioning:
AIHackr should become:
“Competitive intelligence for AI-native SaaS.”

Not:
“BuiltWith for AI.”

Core product promise:
“Track competitor AI stacks, AI feature changes, model/provider shifts, AI messaging, and likely AI cost exposure over time.”

Target users:
Prioritize:
- AI SaaS founders
- Product marketers
- Competitive intelligence teams
- Agencies/consultants doing SaaS audits
- AI infrastructure sales teams

The scanner should not only answer:
“What AI provider does this website use?”

It should answer:
- Is AI visible?
- Is AI likely being used even if provider is hidden?
- What provider, model, SDK, gateway, or infrastructure signals were found?
- What was not found?
- How much of the product was actually observable?
- What does this imply competitively?
- What should the user do next?
- Should this competitor be watched over time?
- What might this AI stack cost at scale?

Build the improved scanning system around multiple scan levels.

Scan levels:

1. Passive Scan
Fast public scan of:
- HTML
- headers
- scripts
- DNS
- static assets
- public JS bundles
- source maps if available
- robots.txt
- sitemap.xml
- metadata
- schema.org
- public API references
- known AI SDK fingerprints

2. Rendered Scan
Use a real browser to:
- load the page
- wait for hydration
- scroll
- capture runtime network calls
- detect loaded scripts
- detect client-side routing
- observe JavaScript bundle behavior
- inspect localStorage/sessionStorage keys
- detect analytics, auth, chat, and AI-related widgets

3. Interactive Probe
Use an automated browser agent to identify and interact with likely AI surfaces:
- buttons like “Ask AI,” “Generate,” “Analyze,” “Summarize,” “Coach me,” “Predict,” “Try demo,” “Chat,” “Upload,” “Plan,” “Recommend”
- textareas and prompt inputs
- search boxes that may trigger AI
- demo flows
- calculators
- onboarding forms
- example prompts
- “try without signup” flows

The probe should perform safe, non-destructive interactions only.

It should capture:
- network requests after interactions
- streaming responses
- WebSocket events
- Server-Sent Events
- long-running generation endpoints
- JSON payloads
- error messages
- endpoint names
- response timing
- request/response headers
- visible UI changes like typing indicators or generated content

4. Authenticated Probe
Paid feature.
Allow the user to manually authenticate or provide a browser session, then run the same interactive probe behind login.

Do not store passwords.
Do not perform destructive actions.
Clearly label authenticated scan evidence as coming from a user-provided session.

5. Watch Mode
Paid feature.
Repeatedly scan selected competitors on a schedule and detect changes over time:
- AI provider changed
- gateway changed
- new AI endpoint detected
- chat/generation surface added
- pricing page changed
- AI marketing copy changed
- model/provider claim changed
- visible AI SDK added or removed
- competitor launched a new AI feature

Include fingerprints for:

Each fingerprint should support:
- endpoint domains
- SDK package names
- script or bundle strings
- request path patterns
- response shape patterns
- error message patterns
- header patterns
- streaming behavior
- gateway signatures
- source-map strings
- confidence weight
- false-positive risk
- evidence category

Detect both direct and indirect AI signals.

Direct signals:
- calls to known AI provider APIs
- visible SDK imports
- gateway URLs
- exposed model names
- known provider headers
- known provider error messages

Indirect signals:
- endpoint paths like /chat, /completion, /generate, /ai, /assistant, /coach, /analyze, /predict, /summarize
- role/content message shapes
- token usage fields
- streaming text/event-stream
- WebSocket token streams
- “thinking” or “typing” states
- regenerate buttons
- prompt templates
- system prompt leakage
- model selection UI
- long-running generation requests
- queueing or job polling
- generated response UI patterns

Add a scoring system.

Scores needed:
1. AI Usage Confidence
How likely is it that the product uses AI?

2. Provider Confidence
How confident are we about the provider?

3. Gateway Confidence
Can we tell whether the app uses direct provider calls, a gateway, a proxy, or server-side abstraction?

4. Visibility Score
How much of the product was observable during the scan?

5. Scan Depth Score
Passive, rendered, interactive, authenticated.

6. Evidence Strength
How strong is the evidence trail?

Important:
If provider confidence is low but AI usage confidence is high, the report should say:
“AI likely used, provider hidden.”

Do not collapse everything into “No public signal.”

Unknown-state logic:
When no provider is detected, classify the result as one of:
- No AI evidence found
- AI likely but provider hidden
- AI possible but not enough interaction observed
- AI behind login likely
- AI hidden behind backend/proxy likely
- Scan blocked or limited
- Probe recommended

Report copy should explain unknowns in plain English.

Example:
“No browser-visible AI provider found. This does not mean the app has no AI. It means we did not observe direct provider calls during this scan. AI may be handled server-side, behind login, through a backend proxy, or only after deeper user interaction.”

Improve the report output.

Every report should include:
1. AI Stack Verdict
2. Executive Summary
3. AI Usage Confidence
4. Provider Confidence
5. Visibility Score
6. What We Found
7. What We Did Not Find
8. Detection Evidence grouped by type
9. Likely Architecture
10. Recommended Next Actions
11. Competitor Comparison CTA
12. Watchlist CTA
13. LLM Margin CTA

Likely Architecture labels:
- Browser-visible provider calls
- Backend-proxied AI
- Gateway-routed AI
- Authenticated-only AI
- Server-side AI likely
- No AI evidence found
- Insufficient interaction depth

Recommended actions should be context-aware.

If no provider detected but AI likely:
- Run AI Probe
- Scan authenticated session
- Compare against competitors
- Add to Watchlist
- Model cost scenarios in LLM Margin

If provider detected:
- Compare provider choice against competitors
- Estimate cost-per-user in LLM Margin
- Add to Watchlist for migration alerts
- Export report

If scan blocked:
- Retry with browser rendering
- Use authenticated probe
- Reduce bot-blocking limitations if possible
- Show which parts of the site were inaccessible
